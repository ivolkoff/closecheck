# Правила анализа и разработки (строгая редакция для LLM)

Этот документ **устанавливает обязательные нормы** для разработки анализатора и тестов. Все пункты ниже имеют **нормативный статус** и подлежат неукоснительному исполнению.

## Ключевые модальные слова и приоритет

* **ОБЯЗАН / ОБЯЗАТЕЛЬНО / ДОЛЖЕН** — *must / shall*: требуется всегда, без исключений.
* **НЕ ДОЛЖЕН / НЕДОПУСТИМО / ЗАПРЕЩЕНО** — *must not / shall not*: полностью запрещено.
* **ТОЧНО СЛЕДОВАТЬ** — выполнять без вольных трактовок и эвристик.
* **ДОПУСТИМО / МОЖНО** — *may*: разрешено, если не противоречит «ОБЯЗАН».
* **ПРИОРИТЕТ**: при любом противоречии с иными документами или исходными текстами **ТОЧНО СЛЕДОВАТЬ** этому документу; данный документ имеет высшую силу.

---

## Базовые принципы

* **Отсутствие зависимостей от имён**

  * Анализатор **ОБЯЗАН** не полагаться на нейминг функций/методов/пакетов/конструкторов **кроме** метода `Close` с точной сигнатурой `func() error`.
  * Любые проверки вида `Shutdown`, `Clean`, `New*`, `Resource`, `*Storage` и т.п. **ЗАПРЕЩЕНЫ**.
  * Единственное допустимое имя — **`Close`**, и только вместе с **жёсткой сигнатурной проверкой**.

* **Типовая опора**

  * Анализатор **ОБЯЗАН** опираться на AST и типовую информацию (`go/types`) и на идентичность объектов.
  * Сопоставления **ДОЛЖНЫ** выполняться через `types.Object` и его `token.Pos`, а не через строковые имена.
  * Сравнения строк имён переменных, типов и путей в логике анализа **НЕДОПУСТИМЫ**.

* **Контейнерные вызовы в `defer`**

  * Вызов `defer X.<метод>()` **НЕ ДОЛЖЕН** считаться очисткой по умолчанию.
  * Он **МОЖЕТ** считаться очисткой **ТОЛЬКО ЕСЛИ** по AST/типовой информации (и, при наличии, межпроцедурной суммаризации тел) **ДОКАЗЫВАЕТСЯ**, что по пути выполнения вызывается `io.Closer.Close()` над **целевым** ресурсом (контейнером или его «дочерними» closers).
  * Идентификация контейнера **ОБЯЗАНА** выполняться через сопоставление получателя (receiver) по `Object.Pos`.

* **Method values**

  * Анализатор **ОБЯЗАН** отслеживать связывания `fn := recv.Method` через `types.Info`.
  * Конструкция `defer fn()` **ДОЛЖНА** трактоваться как вызов метода на `recv`, **НО** засчитывается как очистка **ТОЛЬКО ПРИ ДОКАЗАННОМ** вызове `Close()` по пути выполнения (см. правило выше).

* **Производные ресурсы (ownership/derivation)**

  * Переменные, полученные из контейнера (например, `db := factory.GetDB()`), **ДОЛЖНЫ** помечаться как производные: хранить отображение `varPos -> containerPos`.
  * Поля/значения, полученные от таких переменных (например, `storage.db`), **ДОЛЖНЫ** иметь ссылку на источник (`derivedFromPos`).
  * Закрытие контейнера (любой метод на том же receiver) **ОБЯЗАНО** закрывать и производные ресурсы по pos-цепочке.

* **Межпроцедурные факты (summary)**

  * Экспорт и интерпретация фактов **РАЗРЕШЕНЫ ТОЛЬКО** для `io.Closer.Close()` с точной сигнатурой `func() error`.
  * Любые иные «чистящие» методы по имени **ЗАПРЕЩЕНЫ**.

* **Консервативность доказательств**

  * Если закрытие **НЕЛЬЗЯ ДОКАЗАТЬ** строго по правилам выше, анализатор **НЕ ДОЛЖЕН** засчитывать его. Предпочтение отдаётся корректности доказательства, а не «очевидности» по неймингу.

---

## Реализация и техника

* **Идентичность**

  * **ОБЯЗАТЕЛЬНО** использовать `types.Info.ObjectOf(ident)` и сравнение `Object.Pos()`.
  * Для method-value **ОБЯЗАНО** храниться соответствие `mvVarPos -> recvVarPos`.
  * Для переменных, полученных из фабрики — `varPos -> factoryPos`.
  * Для полей, полученных от переменных — `derivedFromPos`.

* **Обход AST**

  * Глубокая рекурсия/итерация по узлам **ДОПУСТИМА** и при необходимости **ОБЯЗАТЕЛЬНА** для корректности.
  * **ОБЯЗАНО** быть введено предотвращение циклов (upper bound для итераций, мемоизация визитов и т.п.).

* **Сигнатуры**

  * `Close()` **ДОЛЖЕН** считаться валидным **ТОЛЬКО ЕСЛИ** receiver реализует `io.Closer` и сигнатура **ровно** `func() error`.

* **Трассировка/Debug**

  * Любые отладочные принты **ДОЛЖНЫ** быть за флагами (`-print-statements`, `-enable-function-debugger`, и т.п.).
  * «Шумные» логи по умолчанию **ЗАПРЕЩЕНЫ**.

## Запрещено

* Любые эвристики по имени функций/методов/пакетов/конструкторов, **кроме** `Close`.
* Сравнение строк имён переменных, типов и путей для логики анализа.
* Специальные «исключения» под конкретные имена.
* Засчитывать `defer X.<метод>()` как очистку **без** доказательства вызова `Close()` на целевом ресурсе.

---

## Допустимо

* Использовать **только** `Close()` и сигнатуру `func() error` для точек закрытия.
* Использовать `io.Closer` и структурные поля, содержащие `io.Closer`, для определения «закрываемости» значения.
* Применять межпроцедурную суммаризацию **исключительно** для `Close()`.

---

## Репорты и подавление

* Привязка диагностик **ОБЯЗАНА** выполняться к позициям объявлений (pos), а не к местам использования.
* Диагностика в `vendor/` и кэше сборки **ОБЯЗАНА** подавляться (фильтры по файлам, как сейчас).

---

## Производительность и устойчивость

* Приоритет — **корректность**. Если для корректности нужен более глубокий обход, он **ДОЛЖЕН** быть выполнен (с защитами).
* Квадратичные обходы по всему AST **СЛЕДУЕТ ИЗБЕГАТЬ**, если нет строгой необходимости.
* Кеширование результатов **ДОПУСТИМО** там, где это безопасно и улучшает время.

---

## Тесты и образцы

* Тесты **ОБЯЗАНЫ** использовать `analysistest`; новые кейсы — минимальные и самодостаточные.
* Запрещено добавлять зависимости на внешние модули без вендоринга в соответствующий образец.
* Любые изменения анализа **ОБЯЗАНЫ** сохранять стабильность текущих образцов.

## Резюме алгоритма закрытия (нормативное)

1. **Прямой вызов** `io.Closer.Close()` на значении **ДОЛЖЕН** закрывать это значение.

2. `defer X.<метод>()` **МОЖЕТ** засчитываться как очистка **ТОЛЬКО** при доказуемом вызове `Close()` над целевым ресурсом (самим значением или его дочерними closers) по пути выполнения.

3. `defer fn()`, где `fn := X.<метод>` **ДОЛЖЕН** трактоваться как вызов на `X` (по pos-binding method value) и **МОЖЕТ** засчитываться **ТОЛЬКО** при доказанном вызове `Close()` по пути.

4. Возвраты/передачи владения **ДОЛЖНЫ** учитываться **только** через типы/AST и pos-based ownership, **без** опоры на имена.

5. При любой неоднозначности относительно факта закрытия анализатор **НЕ ДОЛЖЕН** засчитывать закрытие.

---

## Чек-лист исполнения (для разработчика/LLM)

**ОБЯЗАН** последовательно выполнить:

1. Получить `types.Info` и маппинги `Object.Pos()` для всех релевантных идентификаторов.
2. Построить/обойти CFG/AST так, чтобы покрыть **все** пути до любой точки выхода.
3. Пометить ресурсы (значения и поля), реализующие `io.Closer`; для производных — заполнить `varPos -> containerPos` и `derivedFromPos`.
4. Для каждого `defer`/method value **ДОЛЖЕН** быть установлен точный receiver по pos; без него — **НЕ СЧИТАТЬ** закрытием.
5. Засчитывать **только** `Close()` с `func() error` на валидном receiver.
6. На каждом `return` и в конце функции проверять состояние всех открытых ресурсов: если нет прямого закрытия и нет активного *доказанного* `defer` — **РЕПОРТ**.
7. Все репорты привязывать к **pos объявления** ресурса.
8. Исключить из анализа `vendor/` и кэш сборки.
9. Любое решение «по имени» или «по очевидности» — **ЗАПРЕЩЕНО**.
10. Если доказать закрытие невозможно — **считать ресурс не закрытым**.

---

## Примечания по особым случаям

* `defer` внутри условных блоков/циклов **НЕ ДОЛЖЕН** считаться универсальной гарантией: он учитывается **только** на путях, где условие выполнено/итерация достигнута.
* `defer` в немедленно вызываемом замыкании **МОЖЕТ** засчитываться (замыкание вызвано на месте), в прочих случаях (горутины, передача наружу) — **НЕ ДОЛЖЕН** засчитываться для текущей функции.
* Возврат ресурса из функции переносит ответственность наверх; текущая функция **НЕ ДОЛЖНА** репортить «утечку» по факту **возврата владения** (без именных эвристик; основание — типы/AST и pos-ownership).
